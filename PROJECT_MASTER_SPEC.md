# Полный свод правил проекта (Master Specification)

**Для кого:** Cursor / Claude Code и архитектор.  
**Цель:** система учёта товаров (закупка → склад → рейс → продажа) с офлайн-режимом для одного клиента.

Подробная продуктовая архитектура и термины заказчика дополнительно зафиксированы в `docs/architecture/`. Этот документ — **операционная библия разработки**: как пишем код, тестируем и синхронизируем офлайн.

---

## Часть 1. Книги, которые лежат в основе

Агенту не нужно читать их целиком. Нужно усвоить принципы.

### 1.1 Чистый код (Роберт Мартин)

| Принцип | Как применяем |
|--------|----------------|
| Имена говорят сами за себя | `remainingKg`, а не `r`; `canShip()`, а не `cs()` |
| Функции делают одно действие | `shipToTrip()` только отгружает, не проверяет документы |
| Комментарии — почему, а не что | Не «отгружаем товар», а «отгрузка до приёмки, чтобы не блокировать склад» |
| DRY | Логику расчёта остатков в одном месте |
| Ошибки, а не коды возврата | `throw new InsufficientStockError()`, а не `return -1` |

### 1.2 Тестирование (Кент Бек / xUnit)

| Принцип | Как применяем |
|--------|----------------|
| Тест до кода | Сначала тест, затем реализация |
| Один тест — одна проверка | Не смешивать несвязанные assert в одном тесте без причины |
| Изолированные тесты | Не зависят друг от друга и от порядка |
| Unit без БД | Репозитории подменяем моками |

### 1.3 Чистая архитектура (Роберт Мартин)

| Принцип | Как применяем |
|--------|----------------|
| Бизнес-логика не знает про БД | `Batch` не импортирует `PrismaClient` |
| Зависимости внутрь | Контроллер → use case → сущность |
| БД и UI — детали | Можно заменить PostgreSQL / фронт, домен не трогаем |

### 1.4 DDD (Эрик Эванс) — термины

| Термин | В проекте |
|--------|-----------|
| Агрегат | `Batch` (партия) — корень |
| Value Object | `Money`, `Weight` |
| Репозиторий | `BatchRepository` — скрывает хранение |
| Ubiquitous Language | `Trip`, `Shortage`, `Receiving` — как у заказчика |

### 1.5 Паттерны (Мартин Фаулер)

| Паттерн | Где |
|--------|-----|
| Transaction Script | Простые сценарии |
| Domain Model | Остатки партии, отчёт по фуре |
| Unit of Work | Группировка операций при синхронизации офлайн-действий |
| Data Mapper | Между БД и доменными объектами |

## Часть 2. Согласование доменной партии с `docs/architecture`

В мастер-правилах для `Batch` изначально были только «склад / в пути / продано / списано». В глоссарии зафиксировано **«Поступление на склад»** как перевод закупки в остаток. Чтобы инварианты сходились без «магии», в модель партии добавлено поле **`pendingInboundKg`**: кг, ещё не оприходованные на склад по данной партии (закуплено/обязательство к поставке — в зависимости от сценария создания).

**Инвариант:**  
`totalKg === onWarehouseKg + inTransitKg + soldKg + writtenOffKg + pendingInboundKg`

Правила в `.cursor/rules/01-domain-batch.mdc` — канон для кода.

**Точность массы:** целевое хранение — **целые граммы** (см. `docs/architecture/data-model/units-and-precision.md`, `.cursor/rules/06-units-and-precision.mdc`). Переход текущего `Batch` с `number` кг — запланированный рефакторинг.

## Часть 5. Чек-лист «ничего не пропустить»

**Перед этапом:** правила обновлены; написаны/запланированы тесты; по сложным местам — вопрос заказчику.

**Перед коммитом:** `npm test` зелёный; нет `any` в новом коде; нет мёртвого закомментированного кода; нет `console.log` кроме помеченных `// DEBUG`.

**Перед релизом:** офлайн сценарии проверены; конфликт даёт понятное сообщение; отчёт сходится на тестовых данных.

## Часть 6. Что сознательно не делаем в первой версии

- Автоматическое разрешение конфликтов офлайн
- Сложные роли (см. roadmap — по мере надобности)
- Регистрация пользователей в продуктовом смысле (если не согласовано отдельно)
- Дашборды и графики
- FIFO по партиям без явного выбора партии
- Интеграция с 1С / кассами

**Отдельно (не в MVP, но запланировано документами):** возвраты от покупателя, полная модель частичных оплат по спорным правилам — только после решения заказчика и обновления `document-map.md` / матрицы прав (см. `risks-and-guardrails.md`).

## Часть 7. Как говорить с агентом (примеры)

**Новый модуль:** прикрепить `@00-master.mdc` и профильный файл (`@01-domain-batch.mdc`), описать сценарии и попросить сначала список тестов, затем код.

**Офлайн-конфликт:** `@02-offline-sync.mdc` + `@03-testing.mdc`, описать начальные остатки и ожидаемый `rejected`.

**Рефакторинг:** `@04-code-quality.mdc` + `@05-architecture-layers.mdc`, указать файл и желаемое разделение слоёв.
