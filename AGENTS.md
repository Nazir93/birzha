# Ориентир для ассистента

Цель — **одна лестница документов**, без размножения «якорных» файлов в `docs/`. Новый важный статус — в существующие файлы ниже, а не отдельный `*-guide.md`, если нет веской причины.

**От чего отталкиваться:** сначала таблицу **«Один указатель: что где искать»** ниже, затем — порядок чтения. Отдельные разделы в `screen-flows.md`, `roles-and-permissions.md` и т.д. **не заменяют** друг друга: у каждого файла своя роль; сводка расхождений — в `screen-flows.md`, раздел «Расхождения, узкие места и принятые решения».

## Контроль приоритетов (чтобы ассистент не путался)

Здесь зафиксировано **кто задаёт направление**: не хаотичные догадки, а документы + явная задача в сообщении.

1. **Главный ориентир по «что делать дальше»** — **`docs/implementation-status.md`**: разделы **«Сводка по этапам»**, **«Очередь работ»**, **«Роли и доступ»**. При фразе пользователя вроде «продолжаем» / «делай что нужно» ассистент **сам выбирает следующий логичный шаг** из этой очереди (один фокус за раз), выполняет и при существенном результате **обновляет** `implementation-status.md` по правилам синхронизации.
2. **Продуктовые границы** — **`PROJECT_MASTER_SPEC.md`** и архитектурные файлы из таблицы «Один указатель». Не расширять scope без основания в доках или прямого указания пользователя.
3. **Спорная бизнес-логика** (долги, возвраты, границы ролей до согласования) — **не придумывать в коде**: см. `docs/architecture/risks-and-guardrails.md`, `roles-and-permissions.md`, `.cursor/rules/07-debts-returns-and-role-edge-cases.mdc`. Нет правила — зафиксировать вопрос в документе или спросить пользователя, а не внедрять «как кажется».
4. **Качество и доверие заказчика** — после значимых изменений **`pnpm check`**; тесты к новой логике; документы синхронизировать по разделу **«Синхронизация правок»** ниже.

## Синхронизация правок (без устаревших дублей)

При изменении смысла обновляй **все затронутые места** из таблицы ниже. Не копируй одни и те же даты, таблицы и формулировки в несколько файлов: **канон** — один файл по теме; в остальных — короткая отсылка.

| Что изменилось | Где обновить (минимум) |
|----------------|-------------------------|
| Права по документам, ячейки матрицы прав | `docs/architecture/processes/roles-and-permissions.md`; если меняется «кто что видит» на экране — также `docs/architecture/ui/screen-flows.md` (матрица панелей и при необходимости «Расхождения…») |
| Панели, домашние экраны, закупщик/кладовщик в UX | `screen-flows.md`; при согласовании с матрицей прав — строка у роли «Закупщик» и таблица в `roles-and-permissions.md` |
| Риск или открытый вопрос (долги, офлайн, возвраты) | `docs/architecture/risks-and-guardrails.md`; если влияет на экраны — строка в `screen-flows.md` («Расхождения…») |
| Факт кода, этап, очередь | `docs/implementation-status.md` |
| Список «куда смотреть» сломался | этот файл — таблица «Один указатель» |

**Дата и подпись согласования матрицы панелей с заказчиком** ведутся **только** в `docs/architecture/ui/screen-flows.md` (блок «Согласование с заказчиком»). В `implementation-status.md` — не дублировать дату: достаточно фразы вроде «матрица панелей согласована» / «в работе» с отсылкой на `screen-flows.md`.

## Единые точки опоры (порядок чтения)

1. **`README.md`** — команды, таблица HTTP API, **стек**, продакшен/CI.
2. **`docs/implementation-status.md`** — **что сделано в коде**, очередь работ, **роли (план внедрения)**, деплой.
3. **`PROJECT_MASTER_SPEC.md`** — принципы, этапы 0–8, ссылки на документы.
4. **`docs/architecture/README.md`** — **индекс** целевой архитектуры (глоссарий, ER, роли, офлайн, риски); не дублирует статус кода — тот в п. 2.
5. **`.cursor/rules/*.mdc`** — правила кода (офлайн, домен, тесты, единицы…).

**Инфраструктура сейчас:** продакшен-сервер **не закуплен** — разработка и проверки **локально** (+ CI на GitHub); VPS не нужен, чтобы продолжать работу в репозитории. Подробности и команды — **`README.md`**, статус — **`docs/implementation-status.md`** (п. 2 в порядке чтения).

## Проверка

**`pnpm check`**; затем при необходимости **`pnpm exec playwright test`**. CI: `.github/workflows/ci.yml`.

## Качество кода и работа без путаницы

Абсолютные принципы уже в **`.cursor/rules/00-master.mdc`** (тесты как контракт, офлайн, единицы, без «на будущее»). Ниже — **как применять** их в монорепо, без дублирования других правил.

### Слои (куда класть изменения)

| Слой | Путь | Правило |
|------|------|---------|
| Домен | `packages/domain` | Инварианты и расчёты; без HTTP и БД. |
| Контракты API/sync | `packages/contracts` | Zod-схемы тел; общие для API и web. |
| Приложение | `apps/api` | Use case → порт → HTTP; бизнес-правила не копировать в роуты «в лоб». |
| Клиент | `apps/web` | UI + вызовы API/sync; валидация форм — `src/validation`, те же схемы что API где возможно. |

Пересекать слои «в обход» (например дублировать расчёт партии в web) — только если явно временно; иначе тесты и правила расходятся с реальностью.

### Перед началом задачи

1. Один смысловой фокус за раз (фича или баг), без примешивания рефакторинга «заодно».
2. Прочитать релевантный кусок **`PROJECT_MASTER_SPEC.md`** и при сомнении — **`docs/architecture`** по таблице «Один указатель».
3. Если меняется поведение для пользователя/ролей/рисков — заранее знать, какие файлы из раздела **«Синхронизация правок»** тронуть после кода.

### После кода (минимум)

1. **`pnpm check`** проходит.
2. Новая логика — **тест** рядом с модулем (Vitest в пакете; для HTTP — существующий стиль `*.test.ts`).
3. Публичный контракт HTTP/sync изменён — обновить **`packages/contracts`** и места парсинга на клиенте.
4. Заметный шаг по продукту — **одна короткая правка** в `docs/implementation-status.md` (раздел «Сделано» или очередь), без дублей дат из других документов.

### Что ухудшает качество (избегать)

- «Удобные» `any`, отключения линтера без причины, копипаст бизнес-формул.
- Скрытые побочные эффекты в use case (логирование ок, изменение состояния «мимо» порта — нет).
- Расширение scope в том же PR без фиксации в документах/статусе.

## Без отдельного ТЗ

Возвраты, частичные оплаты долгов, спорные границы ролей — **`07-debts-returns-and-role-edge-cases.mdc`** и решение заказчика.
