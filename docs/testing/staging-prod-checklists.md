# Чек-листы staging и production

Документ для полного ручного прогона по кабинетам и кнопкам перед релизом.

Связанные артефакты:
- action-catalog: [action-catalog.json](./action-catalog.json)
- золотой сценарий: [golden-scenario.md](./golden-scenario.md)
- нагрузка и безопасность: [load-and-security.md](./load-and-security.md)

## 1) Staging — обязательный полный прогон

### 1.1 Общие проверки
- [ ] `pnpm check` зелёный.
- [ ] Playwright in-memory: `pnpm e2e` зелёный.
- [ ] Playwright auth+pg: `pnpm exec playwright test e2e/role-nav-auth.spec.ts e2e/full-section-regression-auth.spec.ts` зелёный.
- [ ] API PostgreSQL интеграционные тесты зелёные (`TEST_DATABASE_URL`).
- [ ] Smoke/load/security gates зелёные.

### 1.2 Роли и кабинеты
- [ ] Каждая из 8 ролей логинится в корректный кабинет: `admin / manager / purchaser / warehouse / logistics / receiver / seller / accountant`.
- [ ] Прямые URL чужих кабинетов редиректят на домашний кабинет роли.
- [ ] Боковое меню соответствует матрице прав, нет лишних пунктов.
- [ ] Запрещённые действия дают `403` на API (не только скрытие кнопки).

### 1.3 Разделы `/a` (админ)
- [ ] Сводка (`/a`): переходы, KPI, кнопки “Закрыть рейс”.
- [ ] `/a/purchase-nakladnaya`: создать накладную, открыть карточку.
- [ ] `/a/distribution`: выбор склада, отбор, сохранение ПН.
- [ ] `/a/trips`: создать/закрыть/удалить рейс.
- [ ] `/a/assign-seller`: закрепление/смена/открепление продавца.
- [ ] `/a/operations`: фиксация недостачи.
- [ ] `/a/archive`: архив рейсов и документов.
- [ ] `/a/settings/catalog`: создать/удалить склад и калибр.
- [ ] `/a/settings/documents`: редактирование/удаление документов.
- [ ] `/a/settings/team`: создание/роль/удаление пользователя.

### 1.4 Разделы `/o` (операции)
- [ ] Закупка: создать накладную, валидации, ошибки.
- [ ] Погрузка: отбор, сохранение погрузочной, привязка/смена рейса.
- [ ] Рейсы: создание, закрытие, удаление, ограничения.
- [ ] Продавец и продажи: назначение/открепление, ограничения прав.
- [ ] Недостача: запись недостачи и сверка отчёта.
- [ ] Отчёты: выбор рейса, печать, CSV, проверка сумм/масс.
- [ ] Архив: карточки закрытых рейсов и продажи.

### 1.5 Разделы `/s` (продавец)
- [ ] `/s`: весь пошаговый сценарий продажи (канал, рейс, калибр, масса/ящики, оплата, сохранение).
- [ ] `/s/reports`: доступ только к разрешённым отчётам.
- [ ] `/s/archive`: корректные данные по закрытым рейсам.
- [ ] `/s/operations`: ожидаемое поведение по текущей матрице ролей.

### 1.6 Разделы `/b` (бухгалтерия)
- [ ] `/b`: сводка и переходы.
- [ ] `/b/reports`: сверка финансов и масс.
- [ ] `/b/counterparties`: создать/удалить контрагента, проверки прав.

### 1.7 Бизнес-инварианты
- [ ] Золотой сценарий сходится по граммам/копейкам.
- [ ] Оплаты `cash / debt / mixed / card_transfer` корректно отражаются в отчётах.
- [ ] Недостача не превышает допустимое и даёт понятную ошибку.
- [ ] Нет расхождений в остатках между разделами.

## 2) Production — post-deploy чек-лист

### 2.1 Smoke сразу после выката
- [ ] `GET /health` = `200`.
- [ ] Логин admin и seller проходит.
- [ ] Быстрые переходы: `/a`, `/o/reports`, `/s`, `/b`.
- [ ] Service worker/PWA подтянул актуальную версию (без “залипания” старого UI).

### 2.2 Критичный ручной минимум по кнопкам
- [ ] Создать рейс.
- [ ] Создать закупочную накладную.
- [ ] Сформировать погрузочную накладную.
- [ ] Зафиксировать недостачу.
- [ ] Сохранить продажу.
- [ ] Проверить отчёт по рейсу и CSV.
- [ ] Закрыть рейс.

### 2.3 Финальное решение
- [ ] Нет P1/P2 дефектов.
- [ ] Все release-gates в CI зелёные.
- [ ] Заполнен release-report (см. шаблон).
