# Экранные сценарии для web и mobile

## Цель

Описать минимально необходимый набор экранов и пользовательских потоков, чтобы архитектура данных и документов сразу была связана с реальной работой сотрудников.

## Принцип интерфейсов

- `Web Admin` для офисных и управленческих операций
- `Mobile/PWA` для полевых сотрудников
- одна и та же бизнес-логика должна вызываться из обоих интерфейсов

## Роли MVP (первая версия с отдельными панелями)

Черновик для согласования с заказчиком. **Отдельная роль «закупщик» в v1 не выделяется:** привоз на склад и проведение поступления ведёт **кладовщик** в одном контуре (см. раздел ниже).

**Навигация в клиенте (`AppNav`):** первая вкладка — **«Накладная»** (приём на склад); затем «Отчёты и рейсы», «Операции», … После входа пользователи с ролями **кладовщик** и **закупщик** попадают на **`/purchase-nakladnaya`** (остальные — на отчёты или по матрице в `auth/role-panels.ts`). Пользовательский порядок шагов — `docs/guides/dlya-zakazchika-zapolnenie.md`.

| Роль | Входит в панели v1 |
|------|---------------------|
| Администратор | да |
| Кладовщик | да (вкл. фиксацию поступления / приёмки на склад) |
| Логист | да |
| Приёмщик | да (приёмка рейса на месте) |
| Продавец | да |
| Бухгалтер | да |
| Руководитель | да (дашборд и сводные отчёты) |
| Закупщик (отдельно) | **нет в v1** — функции в панели кладовщика |

После согласования заказчиком: **дата и подпись — в блоке «Согласование с заказчиком» ниже**; в [implementation-status.md](../../implementation-status.md) — одна краткая строка статуса (без дубля даты и таблиц). Правило — [`AGENTS.md`](../../../AGENTS.md), раздел «Синхронизация правок».

## Панели по ролям — сводная матрица (роль — клиент — видимость)

Одна точка опоры: **что человек видит первым**, **какие данные обязательны**, **что скрыть** (согласуйте с [roles-and-permissions.md](../processes/roles-and-permissions.md)). Права на документы — там; здесь — UX панелей.

| Роль | Клиент | Домашний экран (после входа) | Обязательные виджеты / журналы | Явно скрыто | Ключевые действия |
|------|--------|------------------------------|----------------------------------|-------------|-------------------|
| **Администратор** | Web | Панель администрирования | Пользователи, роли, справочники (или ссылки), аудит / конфликты синка (когда будет) | — | CRUD пользователей и ролей, настройка системы |
| **Кладовщик** | Web (+ при необходимости PWA) | **Накладная** (`/purchase-nakladnaya`) после входа — приём на склад; далее операции и отчёты | Остатки **своего** склада, закупки/накладные к приёмке, партии, задания на отгрузку в рейс | Чужие склады, финансовые отчёты не своего уровня | Ввести накладную (поступление), при необходимости распределение вне/в системе, отгрузка в рейс |
| **Логист** | Web | Рейсы | Список рейсов, статусы, машина/водитель/маршрут, отгрузки | Продажи, касса, долги (кроме просмотра по рейсу по политике) | Создать/вести рейс, контроль отгрузки и приёмки |
| **Приёмщик** | PWA (приоритет) | Мои рейсы к приёмке | Отгруженные позиции по рейсу, расхождения, фото | Закупочные цены, чужие рейсы вне зоны | Приёмка рейса, недостача/порча, синхронизация |
| **Продавец** | PWA | Остаток / «на машине» | Свой остаток по партиям, быстрая продажа, долги **своих** клиентов (в рамках политики) | Закупочные цены (если не разрешено), чужие продажи, склад Дагестана | Продажа, долг, оплата долга (если разрешено политикой), списание своего остатка, офлайн-очередь |
| **Бухгалтер** | Web | Финансы и сверки | Все продажи, дебиторка, оплаты, отчёты по деньгам | Редактирование товарных количеств в продажах | Сверка, подтверждение оплат долга (по матрице), фин. отчёты |
| **Руководитель** | Web | Дашборд | Агрегаты: остатки, рейсы, продажи дня, долги, при необходимости — расхождения | — | Просмотр и утверждение по политике, без массовой правки первички |

**Замечание по долгам:** кто вносит оплату долга (продавец vs только бухгалтер) — **одно бизнес-правило**; до согласования колонки «Ключевые действия» для продавца/бухгалтера уточняются (см. [risks-and-guardrails.md](../risks-and-guardrails.md), риск 2).

## Решение по процессу закупки и складу (v1)

- В полевом процессе заказчика **кладовщик** принимает товар на склад и **фиксирует поступление** (количество, партия, цена закупки для учёта).
- В MVP **не вводится отдельная панель «закупщик»**; сценарии «закупочная накладная → поступление» доступны **в панели кладовщика** (или общий контур «Склад»). При появлении отдельного закупщика позже — вынести в отдельную роль и обновить матрицу в [roles-and-permissions.md](../processes/roles-and-permissions.md).

## Расхождения, узкие места и принятые решения

Ниже — **сводка**: где документы или код расходятся с целевой картиной, в чём риск и что уже решено или отложено. Детали по техническим рискам — в [risks-and-guardrails.md](../risks-and-guardrails.md); по статусу кода — в [implementation-status.md](../../implementation-status.md).

| Тема | Расхождение или узкое место | Решение / статус |
|------|-----------------------------|------------------|
| **Продукт vs код** | В [implementation-status.md](../../implementation-status.md) — партии, рейсы, отчёт, sync, **закупочная накладная**; **вход и фильтр панелей по ролям** — в **`apps/web`** (`auth/role-panels.ts`, маршруты с `RequirePanel`). | Форма накладной: вкладка **«Накладная»** (`/purchase-nakladnaya`, панель `nakladnaya`); ссылка с «Операций». Отдельный домашний экран «только склад» по матрице ниже — по итерациям. Серверные права как у создания партии (`batchCreate`). |
| **Матрица прав vs матрица панелей** | [roles-and-permissions.md](../processes/roles-and-permissions.md) задаёт **документы**; этот файл — **что видно на экране**. Можно потерять связь. | Две таблицы дополняют друг друга: права на сервере обязательны; панель — UX. При сомнении сначала право, потом скрытие в UI. |
| **Роль «Закупщик»** | В матрице прав есть колонка «Закупщик»; в v1 **отдельной панели нет**. | **Принято для v1:** закупочная накладная и поступление в **одном контуре с кладовщиком**; колонка в матрице прав сохранена для будущего разделения. |
| **Названия разделов ниже по файлу** | Блоки «Mobile/PWA **для закупщика**» и «**для кладовщика**» описывают сценарии, которые в v1 могут **сходиться в одной панели**. | **Принято:** не удалять сценарии — это готовые экраны; при реализации v1 показывать их **в панели кладовщика** (или переименовать группу в UI позже). |
| **Оплаты долга (продавец vs бухгалтер)** | Разные сценарии в поле дают расхождение дебиторки и реальности. | **Открыто до заказчика** — [риск 2 в `risks-and-guardrails.md`](../risks-and-guardrails.md); после выбора **одного** варианта — обновить матрицу прав и строку продавца в таблице панелей выше. |
| **Экран «Оплата долга» у продавца** | В сценариях ниже экран есть; в панели продавца действие может быть **запрещено** политикой. | Показывать/разрешать экран **только** в соответствии с решением по риску 2; иначе скрыть действие, оставить просмотр долга при необходимости. |
| **Частичные оплаты долгов** | Упомянуты в дорожной карте; модель «как в банке» усложняет роли. | **После** базовой матрицы прав и правил заказчика; не блокирует пилот с простыми оплатами — см. [implementation-status.md](../../implementation-status.md). |
| **Возвраты** | В списке офисных экранов есть «возвраты»; в домене сущности **ещё нет**. | **Не симулировать** возврат правками остатка; [риск 3 в `risks-and-guardrails.md`](../risks-and-guardrails.md) — после базового контура. |
| **Офлайн: две продажи подряд** | Локальный остаток устаревает до ответа сервера. | [Риск 1 в `risks-and-guardrails.md`](../risks-and-guardrails.md): после успешного sync — подтягивать актуальные остатки; при `rejected` — стоп очереди и понятное сообщение. |
| **POST /sync и роль** | JWT допускает любую MVP-роль на вход в `/sync`, но действия разные (продажа vs отгрузка и т.д.). | При **`REQUIRE_API_AUTH`** сервер проверяет роль на **`actionType`** (как у соответствующих REST-операций); при запрете — **`sync_forbidden`** в теле ответа (`rejected`, не HTTP 403), чтобы очередь на клиенте остановилась с текстом, а не как «сеть». См. [implementation-status.md](../../implementation-status.md), HTTP API. |
| **Очередь на общем устройстве** | Один браузер, несколько учёток — риск смешать офлайн-действия и `deviceId` для идемпотентности. | В **`apps/web`** при включённом входе область хранения очереди и **`deviceId`** привязаны к **`user:<id>`** (отдельная IndexedDB / ключи); без входа — прежняя общая область `default`. |
| **Конфликт синхронизации** | Два устройства спорят за один остаток. | Сервер отклоняет с причиной; **без автослияния**; очередь на клиенте останавливается — см. [risks-and-guardrails.md](../risks-and-guardrails.md) и офлайн-правила. |
| **Кг в Excel vs система** | Сумма дробных кг «не сходится». | Канон — **целые граммы** в БД; кг — отображение ([риск 4 в `risks-and-guardrails.md`](../risks-and-guardrails.md), [`units-and-precision.md`](../data-model/units-and-precision.md)). |
| **Клиент в отчёте** | До справочника контрагентов нужна хоть какая-то подпись в продажах. | **Сейчас:** свободное поле **`clientLabel`** (до 120 симв.) в продаже с рейса; в отчёте — агрегат **`sales.byClient`**. Позже можно заменить на ID клиента без ломки партий. |
| **Тесты под «панели по ролям»** | Нужен браузерный сценарий с JWT. | **`e2e/role-nav-auth.spec.ts`** (бухгалтер vs кладовщик, видимость вкладок): PostgreSQL, см. **`README.md`**, `pnpm e2e:roles`. Плюс обязательный **`pnpm check`** и дым **`e2e/golden-smoke.spec.ts`** (in-memory API). |
| **VPS / PostgreSQL** | Интеграционные тесты и пилот удобнее на реальной БД. | **Следующий этап** после согласования матрицы панелей; не блокирует доработку документов. |

**Правило ведения:** при новом спорном месте добавлять строку в таблицу выше и при необходимости — пункт в [risks-and-guardrails.md](../risks-and-guardrails.md). Общее правило не дублировать правки между файлами — [`AGENTS.md`](../../../AGENTS.md), раздел «Синхронизация правок».

## Сквозные UI-правила

- в каждом журнале нужен статус синхронизации
- все документы должны показывать, проведены ли они на сервере
- при конфликте пользователь должен видеть причину и рекомендуемое действие
- остатки должны обновляться после каждого подтвержденного действия

## Обязательные уведомления

- товар принят по рейсу
- продажа синхронизирована
- конфликт синхронизации
- превышение лимита долга
- закрытие рейса
