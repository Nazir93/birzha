# Как работает система по ролям (сценарии заказчика)

Документ снимает разрыв между «человеческим» описанием и `roles-and-permissions.md`. При противоречии с матрицей прав — **сначала уточнить у заказчика**, затем обновить оба файла.

## Общая цепочка движения товара

```text
Закупщик → Склад (Манас / Каякент) → Рейс (фура) → Приёмка (Москва / регионы) → Продавец → Покупатель
```

Система фиксирует **каждый** шаг; остатки и деньги не должны «пропадать» без документа.

---

## Роль 1 — Закупщик

**В реальности:** закуп у поставщиков, бумажная накладная.

**В системе:**

1. Создаёт **закупочную накладную** (веб или моб.): поставщик, товар, **кг и ящики**, цена за кг, целевой склад, **фото** накладной.
2. Видит статус вроде «ожидает поступления на склад».

**Не делает:** приёмку на склад, продажу; чужие закупки — по политике (обычно только свои, руководитель — шире).

---

## Роль 2 — Кладовщик

**В реальности:** приём машины, пересчёт, брак.

**В системе:**

1. Список закупок, ожидающих приёмку.
2. **Поступление на склад:** фактические кг/ящики, брак/списание при приёмке.
3. **Распределение по направлениям** (Москва / регионы и т.д.) — если включено в MVP.
4. **Отгрузка в рейс:** выбор рейса, количество кг, подтверждение.

**Видит:** остаток на складе до/после операций в своей зоне ответственности.

---

## Роль 3 — Логист

**В реальности:** машины, водители, маршруты.

**В системе:**

1. Создание **рейса**: идентификация фуры/номер, водитель, маршрут, даты.
2. Статусы рейса: формируется → загружен → в пути → прибыл → закрыт (уточнять точный набор в `document-map.md`).
3. **Отчёт по фуре** после закрытия: отгружено / принято / недостача / продано / остаток / выручка / расходы / прибыль — цифры должны **сходиться** (см. `docs/testing/golden-scenario.md`).

**Не делает:** продажи, приём денег.

---

## Роль 4 — Приёмщик

**В реальности:** встреча фуры, пересчёт, фиксация недостачи/порчи.

**В системе:**

1. Список прибывших рейсов, ожидающих приёмку.
2. **Приёмка:** факт кг, расхождения, фото повреждений.
3. После приёмки — распределение на продавцов по правилам продукта (авто/ручное — уточняется в roadmap).

**Видит:** очередь приёмок и историю, чтобы не принять дважды.

---

## Роль 5 — Продавец (критичная роль, PWA, часто офлайн)

**В реальности:** продажа с телефона, наличные / долг.

**В системе:**

1. **Остаток** по выданным партиям (без лишних чужих данных — см. политику видимости закупочной цены).
2. **Продажа:** товар, кг, тип (опт/розница), цена продажи, оплата (нал / долг / смешанная).
3. **Долг:** идентифицированный клиент, сумма долга в отчётах.
4. **Офлайн:** действие сохраняется локально → синхронизация при сети; при конфликте — **отклонение с понятным текстом**, очередь стоп (см. `.cursor/rules/02-offline-sync.mdc`).

**Не видит (по умолчанию):** закупочные цены (если руководитель не разрешил), чужие продажи, склад Дагестана.

---

## Роль 6 — Бухгалтер

**В системе:**

1. Продажи, фильтры (опт/розница, нал/долг, продавец).
2. **Долги и оплаты:** частичные погашения, остаток долга.
3. Финансовые отчёты, сверка с продавцом.

**Не делает:** складские операции, продажу товара как кассира на рынке (уточнять границу с продавцом).

---

## Роль 7 — Руководитель

**В системе:** дашборд (остатки, рейсы в пути, продажи дня, долги, прибыль), отчёт по фуре, по продавцам; справочники и пользователи — по политике.
