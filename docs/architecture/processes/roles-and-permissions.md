# Роли, права и зоны ответственности

**Сводка планов по кабинетам** — [`../cabinets.md`](../cabinets.md) (префиксы `/a`, `/o`, `/s`, `/b` и защита `RequireCabinet`).

**Реализация в веб** — `apps/web/src/auth/role-panels.ts` и маршруты в `App.tsx`. **Справочники «склад»/«калибр» (создание и удаление):** в UI — раздел `/a/inventory` (роль `admin`); на API — `inventoryCatalogWrite` в `apps/api/src/http/route-auth.ts`. **Закуп и кладовщик** выбирают склад/калибр в накладной, но **не** создают/удаляют эти сущности через `POST/DELETE` при включённой авторизации.

Ориентир ассистента: [`AGENTS.md`](../../../AGENTS.md).

## Принцип доступа

Права выдаются по 4 измерениям:

1. `Роль`
2. `Подразделение или склад`
3. `Тип документа`
4. `Действие` — просмотр, создание, подтверждение, проведение, отмена, экспорт

## Основные роли

### Админ
Полный доступ к настройкам, справочникам, пользователям, ролям и аудиту.

### Руководитель
Доступ ко всем отчетам, остаткам, рейсам, закупкам, продажам и долгам. Может утверждать ключевые документы, но не должен массово редактировать первичку.

### Закупщик
Создает закупочные накладные, прикладывает фото документов, видит свои закупки и связанные поступления.

В MVP **отдельной панели «закупщик» нет:** сценарии закупки и поступления на склад — в одном контуре с **кладовщиком**; целевой план **отдельных кабинетов** — [`../cabinets.md`](../cabinets.md). Колонка «Закупщик» в матрице ниже — для будущего разделения.

### Кладовщик
Подтверждает поступления, распределяет товар, формирует отгрузки в рейс, проводит инвентаризацию склада.

### Логист
Создает и ведет рейсы, назначает машину и водителя, контролирует статусы отгрузки и приемки.

### Приемщик
Подтверждает приемку рейса, фиксирует расхождения, прикладывает фото повреждений.

### Продавец
Принимает свой товар, видит остатки, оформляет продажи, долги, оплаты долгов и списания в пределах своих прав.

### Бухгалтер
Контролирует оплаты, дебиторку, расходы, сверки и финансовые отчеты.

## Ограничения по контексту

### Склады
- закупщик видит только связанные с ним закупки
- кладовщик работает только со своим складом
- руководитель и админ видят все склады

### Рейсы
- продавец видит только рейсы, где он назначен приемщиком или продавцом
- приемщик видит только активные рейсы своей зоны
- логист видит все рейсы в своей ответственности

### Продажи и долги
- продавец видит только свои продажи и долги
- бухгалтер видит все продажи и все оплаты

**Ограничение по складу (scoped роли):** у пользователя с ролями `warehouse` или `receiver` и `scope_type = warehouse`, `scope_id = <id склада>` списки **`GET /api/batches`** и **`GET /api/purchase-documents`** содержат только строки этого склада; чужой документ на **`GET /api/purchase-documents/:id`** — ответ **403**. См. `apps/api/src/auth/warehouse-scope.ts`.

**Закупщик и накладные:** при создании накладной в PostgreSQL сохраняется **`created_by_user_id`** (JWT `sub`). Учётная запись с глобальной ролью **`purchaser`** в списке и карточке накладной видит только документы **без автора** (наследие до миграции) или **своего** автора. См. `apps/api/src/auth/purchase-scope.ts`.

**Рейс и полевой продавец:** у рейса есть **`assigned_seller_user_id`** (кто закреплён «в поле»). Для **`GET /api/trips`** и доступа к **`GET /api/trips/:id`** / отчёту учётная запись с **только** полевой ролью `seller` видит только рейсы, закреплённые именно за ней; неназначенные и чужие рейсы скрыты/дают **403**. Назначение выполняется отдельным действием **`POST /api/trips/:tripId/assign-seller`** (“Отгрузить с рейса”) с ролями `admin`, `manager`, `purchaser`, `logistics`.

**Реализация в API (MVP, текущий код):** в таблицу `trip_batch_sales` пишется `recorded_by_user_id` (пользователь из JWT при `POST /batches/…/sell-from-trip` и `POST /sync` с `sell_from_trip`). Для учётной записи с **глобальной** ролью **только** `seller` (без `warehouse`, `logistics`, `manager` и т.д., см. `isGlobalSellerOnly` / `isFieldSellerOnly`) блоки **`sales`** и связанные **`financials`** в **`GET /api/trips/:tripId/shipment-report`** строятся только по таким строкам; **отгрузка в рейс** и **недостача** в отчёте — общие по рейсу. Остальные роли получают полный отчёт по продажам.

**Назначение продавца на рейс:** список кандидатов — **`GET /api/trips/field-seller-options`** (роли `admin`, `manager`, `purchaser`, `logistics`), активные пользователи с глобальной ролью `seller`.

- руководитель видит агрегаты и детализацию по ролям и регионам

## Разделение действий

### Создание
Пользователь может собрать черновик документа, но не проводить его без соответствующего права.

### Подтверждение
Подтверждение означает, что пользователь отвечает за корректность данных до проведения.

### Проведение
Проведение создает движения и должно быть доступно только ролям, которые реально управляют остатками и деньгами.

### Отмена
Отмена проведенного документа выполняется только через обратный документ или специальный административный сценарий с записью в аудит.

## Открытый вопрос (согласовать с заказчиком)

Частичные оплаты долгов и право **продавца** вносить оплату vs только **бухгалтер**: в тексте сценариев заказчика и в матрице выше возможны разные акценты. Нужно **одно** бизнес-правило; до согласования см. `../risks-and-guardrails.md` (риск 2).

## Критические запреты

- продавец не может вручную увеличивать свой остаток
- закупщик не может проводить складские документы
- приемщик не может менять уже закрытый рейс
- бухгалтер не должен редактировать товарные количества в продажах
- документ с созданными движениями не должен редактироваться без следа в аудите
