# Дорожная карта внедрения

## Цель

Разделить полную архитектуру на очереди внедрения так, чтобы:
- не потерять целостность модели
- запустить рабочую систему быстро
- не переписывать фундамент на втором этапе

## Базовый принцип внедрения

Сразу проектируется вся доменная модель, но в разработку идет только тот набор модулей, который нужен для первого рабочего оборота товара.

## Очереди внедрения

```mermaid
flowchart LR
    phase0[Этап0_УточнениеПредметнойОбласти] --> phase1[Этап1_БазовыйУчет]
    phase1 --> phase2[Этап2_ПродажиИДолги]
    phase2 --> phase3[Этап3_ОтчетностьИКонтроль]
    phase3 --> phase4[Этап4_РасширениеИСтабилизация]
```

## Этап 0. Подготовка к разработке

### Результат
Команда согласовала термины, документы, роли и базу данных.

### Что должно быть готово
- словарь терминов
- список ролей
- модель документов
- ER-модель
- список справочников
- список отчетов

### Критерий завершения
- нет спорных трактовок по партиям, долгам, приемке и остаткам

## Этап 1. Базовый учет товара

### Цель
Запустить путь `закупка -> склад -> отгрузка -> приемка`.

### Модули
- пользователи и роли
- справочники товаров и складов
- закупочная накладная
- поступление на склад
- партии
- складские движения
- распределение по направлению
- рейсы
- отгрузка в рейс
- приемка рейса

### Что пока не включать
- сложные возвраты
- инвентаризацию по всем контекстам
- расширенную аналитику прибыли

### Критерий завершения
- можно зафиксировать закупку
- можно увидеть остаток на складе
- можно загрузить товар в фуру
- можно принять товар в Москве или регионе

## Этап 2. Продажи и долги

### Цель
Запустить путь `остаток продавца -> продажа -> долг -> оплата`.

### Модули
- остаток продавца
- продажи
- продажа в долг
- клиенты
- оплаты долгов
- денежные движения
- базовые списания продавца
- мобильный офлайн-контур для продавца

### Критерий завершения
- продавец видит свой остаток
- продавец делает продажи офлайн и онлайн
- офис видит, кто сколько продал
- офис видит сумму долга и оплаты

## Этап 3. Отчетность и контроль

### Цель
Дать руководителю и бухгалтерии итоговую картину по рейсам, остаткам и дебиторке.

### Модули
- отчет по фуре
- остатки по складам
- остатки по продавцам
- продажи по продавцам и клиентам
- долги и просрочка
- расходы по рейсам
- аудит действий
- журнал конфликтов синхронизации

### Критерий завершения
- по каждому рейсу видны загрузка, приемка, продажи, остаток, долги и расходы

## Этап 4. Расширение и стабилизация

### Цель
Углубить контроль и подготовить систему к масштабированию.

### Модули
- распределение по качеству с подпартиями
- возвраты
- передача между продавцами
- инвентаризация
- расширенная прибыль по партиям
- дополнительные роли и ограничения
- оптимизация производительности

### Критерий завершения
- система покрывает нестандартные сценарии без ручного Excel-обхода

## Состав MVP

MVP включает только то, без чего нельзя пройти основной цикл бизнеса.

### Входит в MVP
- авторизация и роли
- товары, варианты, склады, рынки, клиенты
- закупочная накладная
- поступление на склад
- партии
- складские движения
- распределение по направлению
- рейсы
- отгрузка в рейс
- приемка рейса
- остаток продавца
- продажа
- продажа в долг
- оплата долга
- расход по рейсу
- отчет по фуре
- офлайн-продажи и офлайн-приемка

### Не входит в MVP
- сложные возвраты
- передача между продавцами
- полная инвентаризация
- глубокая аналитика прибыли по каждой подпартии

## Ключевые риски внедрения

### Бизнес-риски
- разные сотрудники по-разному понимают качество и направление
- продавцы могут сопротивляться вводу данных на рынке
- часть клиентов по долгам может быть не идентифицирована

### Технические риски
- офлайн внедрен слишком поздно
- остатки считаются не из движений, а из случайных обновлений
- отчеты строятся из отдельных таблиц, не связанных с первичкой

### Организационные риски
- нет ответственного за справочники
- нет дисциплины по закрытию рейсов
- нет процесса разбора расхождений

## Обязательные решения до старта кодинга

- утвердить список статусов документов
- утвердить список ролей
- утвердить справочник качества и направлений
- выбрать стек web/mobile/backend
- выбрать формат офлайн-клиента: нативный mobile или PWA
- определить, кто владелец продукта и кто принимает бизнес-решения

## Минимальная команда на запуск

- 1 product owner от бизнеса
- 1 backend developer
- 1 frontend web developer
- 1 mobile/PWA developer
- 1 QA или сильный ручной тестировщик
- 1 аналитик или архитектурный владелец требований

## Определение готовности MVP

MVP считается готовым, если:
- все основные документы проходят полный цикл
- продавец может работать при плохом интернете
- офис видит остатки, продажи, долги и отчет по фуре
- расхождения по приемке не теряются
- данные не нужно вручную переписывать в отдельные таблицы
