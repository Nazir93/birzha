# Кабинеты и отдельные входы (направление продукта)

**Черновик** — сюда сходится план, без дублирования детальных сценариев. Детальная матрица прав: [`processes/roles-and-permissions.md`](processes/roles-and-permissions.md).

## Целевая схема

| Контур | Кто | Вход / клиент (цель) |
|--------|-----|----------------------|
| **Админ** | Владелец, админ, (опц.) руководство | **Отдельная админ-панель** — единая точка контроля, настройка пользователей, ролей, справочников, аудит, всё что копится по пути. В текущем репозитории часть этого — в веб-разделе под ролью `admin` + API. |
| **Закуп / склад / рейс** | Закуп, кладовщик, логист, приёмщик (сценарий приём–распределение–отгрузка) | **Отдельное приложение (или выделенный веб-контур)** — накладная, распределение, операции, рейсы, офлайн по мере готовности. |
| **Полевые продажи** | Продавцы | **PWA (отдельный билд/раздел)** — кассы, остатки, продажи, очередь sync, без тяжёлой админки. |
| **Бухгалтерия** | Бухгалтер | **Отдельный кабинет/раздел** — деньги, сверки, дебиторка, без ручки товарного количества, если иначе не согласовано. |

## Вход

У каждого рода работ — **свой вход** (свой URL, билд или конфиг PWA, отдельная сессия/приложение), чтобы не мешать роли в одной «куче». Реализация пошагово: в коде сейчас **один** `apps/web` + матрица панелей `role-panels` и `/login` — дальше вынос в отдельные приложения/поддомены по мере плана, без «большой переписи в один день».

## С чем сверяться

- **Код панелей:** `apps/web/src/auth/role-panels.ts`, маршруты с `RequirePanel` в `App.tsx`.
- **API и права:** `apps/api/src/http/route-auth.ts` — сервер нельзя «обойти» только скрытием меню.
